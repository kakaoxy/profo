# Project 模块产品文档

## 一、模块概述

### 1.1 产品定位

Project 模块是 ProFo 房产数据仓库系统的核心业务模块，负责管理房源资产的全生命周期。从业主签约、装修改造、房源上架到最终售出，Project 模块提供完整的项目跟踪、状态管理和数据分析能力。

### 1.2 核心目标

- **全生命周期管理**：覆盖房源从签约到售出的完整业务流程
- **财务透明化**：实时追踪项目收入、支出和 ROI 计算
- **状态可视化**：清晰展示项目在各阶段的进度和关键节点
- **数据驱动决策**：通过监控和分析功能辅助业务决策

### 1.3 业务价值

| 价值维度 | 具体体现             |
| ---- | ---------------- |
| 效率提升 | 统一的项目管理界面，减少信息分散 |
| 风险控制 | 完整的状态流转记录，防止业务漏洞 |
| 财务管控 | 精确的现金流追踪和 ROI 计算 |
| 决策支持 | 多维度数据统计和趋势分析     |

### 2.3 功能详细说明

#### 2.3.1 项目管理

**项目创建**

- 支持多 Tab 表单：基础信息、代理协议、业主信息、附件上传
- 自动保存草稿功能
- 表单验证与错误提示
- 项目名称自动生成（小区名称 + 地址）

**项目列表**

- 状态筛选：全部、签约、装修、在售、已售
- 搜索功能：支持小区名称模糊搜索
- 分页展示：默认每页 20 条
- 快捷操作：查看详情、编辑、删除

**项目详情**

- 侧边栏展示，支持多视图切换
- 根据项目状态自动切换视图模式
- 实时数据刷新

#### 2.3.2 装修管理

**阶段跟踪**

- 8 个标准阶段：拆除、设计、水电、木瓦、油漆、安装、交付、已完成
- 阶段完成日期记录
- 阶段照片上传和管理

**费用管理**

- 硬装合同金额记录
- 4 个支付节点配置
- 软装预算与实际成本
- 其他费用：设计费、拆旧费、垃圾清运费

#### 2.3.3 销售管理

**房源上架**

- 挂牌日期和价格设置
- 自动转入在售状态

**客户跟进**

- 跟进方式：电话、微信、面谈、实地带看
- 跟进记录时间轴

**成交管理**

- 成交价和成交日期记录
- 销售人员角色分配：渠道负责人、房源维护人、联卖谈判人
- 出价记录管理

#### 2.3.4 财务管理

**收支记录**

- 收入类型：售房款、溢价款、服务费、回收保证金、其他收入
- 支出类型：履约保证金、中介佣金、装修费、营销费、税费、运营费、其他支出

**财务统计**

- 总收入、总支出自动计算
- 净现金流 (Net Cash Flow)
- ROI 投资回报率计算

### 3.4 数据库设计

#### 3.4.1 核心表结构

**projects 表（项目主表）**

```sql
CREATE TABLE projects (
    id VARCHAR(36) PRIMARY KEY,
    name VARCHAR(700) NOT NULL COMMENT '项目名称',
    community_name VARCHAR(200) NOT NULL COMMENT '小区名称',
    address VARCHAR(500) NOT NULL COMMENT '物业地址',
    area NUMERIC(10,2) COMMENT '产证面积(m²)',
    layout VARCHAR(50) COMMENT '户型',
    orientation VARCHAR(50) COMMENT '朝向',
    status VARCHAR(20) NOT NULL COMMENT '项目状态',
    renovation_stage VARCHAR(20) COMMENT '改造子阶段',
    is_deleted BOOLEAN DEFAULT FALSE COMMENT '逻辑删除标记',
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL
);
```

**project\_contracts 表（合同表）**

```sql
CREATE TABLE project_contracts (
    id VARCHAR(36) PRIMARY KEY,
    project_id VARCHAR(36) NOT NULL UNIQUE COMMENT '项目ID',
    contract_no VARCHAR(100) COMMENT '合同编号',
    signing_price NUMERIC(15,2) COMMENT '签约价格(万)',
    signing_date DATETIME COMMENT '签约日期',
    signing_period INTEGER COMMENT '合同周期(天)',
    extension_period INTEGER COMMENT '顺延期(天)',
    extension_rent NUMERIC(15,2) COMMENT '顺延期租金',
    cost_assumption VARCHAR(50) COMMENT '税费承担方',
    planned_handover_date DATETIME COMMENT '交房时间',
    other_agreements TEXT COMMENT '其他约定',
    signing_materials JSON COMMENT '合同附件',
    contract_status VARCHAR(20) DEFAULT '生效',
    is_deleted BOOLEAN DEFAULT FALSE
);
```

**project\_sales 表（销售表）**

```sql
CREATE TABLE project_sales (
```

## 七、使用指南

### 7.1 快速开始

#### 7.1.1 创建第一个项目

1. 进入项目管理页面
2. 点击右上角"新建项目"按钮
3. 填写必填字段：
   - 小区名称（用于生成项目名称）
   - 物业地址（用于生成项目名称）
   - 合同编号
4. 点击"创建项目"完成

#### 7.1.2 项目状态流转

| 当前状态 | 可执行操作  | 下一状态 |
| ---- | ------ | ---- |
| 签约   | 交房开始装修 | 装修   |
| 装修   | 装修完成上架 | 在售   |
| 在售   | 标记已售   | 已售   |

### 7.2 最佳实践

#### 7.2.1 数据录入规范

- **小区名称**：使用标准名称，便于后续统计
- **合同编号**：保持唯一性，建议包含年份和序号
- **日期格式**：统一使用 YYYY-MM-DD 格式
- **金额单位**：签约/成交价格使用"万元"，其他费用使用"元"

#### 7.2.2 附件管理

- 合同附件支持类型：PDF、图片、Word、Excel
- 建议按类别上传：身份证、房产证、合同扫描件
- 单文件大小限制：10MB

#### 7.2.3 装修阶段管理

- 及时上传各阶段照片，形成完整的蜕变影像
- 准确记录阶段完成日期，便于进度追踪
- 装修费用及时录入，确保财务数据准确

### 7.3 常见问题

#### Q1: 项目名称是如何生成的？

项目名称由系统自动生成，格式为：`{小区名称} - {地址}`。创建项目后如需修改，可编辑项目信息。

#### Q2: 已售状态的项目还能修改吗？

已售项目只能修改基础信息（小区名称、地址、面积等），不能修改合同、销售等核心财务数据。

#### Q3: 如何查看项目的 ROI？

ROI 根据财务流水自动计算：

- 公式：`ROI = (总收入 - 总支出) / 总支出 × 100%`
- 在项目列表和详情页均可查看

#### Q4: 状态流转有权限限制吗？

状态流转需要相应的操作权限，具体权限配置请联系系统管理员。

#### Q5: 装修阶段可以跳过吗？

系统支持 8 个标准阶段，但实际项目中可以按需标记完成，不必严格按照顺序。

#### Q6: 如何导出项目数据？

点击项目列表页面的"导出"按钮，可导出当前筛选条件下的项目数据（Excel 格式）。

## 八、技术规范

### 8.1 代码规范

#### 8.1.1 命名规范

- **Python**：snake\_case（函数、变量）、PascalCase（类）
- **TypeScript**：camelCase（函数、变量）、PascalCase（接口、类型）
- **文件命名**：kebab-case（组件）、snake\_case（后端模块）

#### 8.1.2 类型要求

- Python：所有函数参数和返回值必须添加类型提示
- TypeScript：禁止使用 `any` 类型，必须明确定义接口

#### 8.1.3 文件大小

- 单个代码文件不超过 250 行，超出需按逻辑拆分

### 8.2 API 规范

- 成功响应直接返回 Pydantic 模型，不使用 `code`/`msg`/`data` 包装
- 错误响应使用 FastAPI 标准 HTTPException，格式为 `{"detail": "错误信息"}`
- 列表接口统一返回 `items`/`total`/`page`/`size` 结构
- 状态码规范：GET/PUT 成功返回 200，POST 成功返回 201，DELETE 成功返回 204

### 8.3 数据库规范

- 状态字段使用字符串枚举，不使用整数
- 核心业务数据使用逻辑删除（`is_deleted` 字段）
- 表和字段必须添加注释说明业务含义
- 跨层级关联使用软引用，不设置强制外键约束

# ProFo 房地产翻新与销售管理系统 - 完整项目指南

## 一、项目背景与目标

### 1.1 项目定位
- **项目名称**: ProFo 房地产翻新与销售管理系统
- **核心角色**: 负责项目的资深全栈架构师
- **项目使命**: 维护并演进一个房地产翻新与销售全流程管理系统
- **核心理念**: 数据完整性、严格类型定义、领域解耦
- **核心架构策略**: 解耦 + 写时复制（Decoupled + Copy-on-Write）

### 1.2 项目目标
1. 构建覆盖房地产翻新与销售全流程的管理系统
2. 实现L1-L4四层业务领域的清晰解耦
3. 建立严格的数据完整性保障机制
4. 提供完整的类型安全前后端开发体验
5. 支持从开发到生产的无缝部署

### 1.3 核心价值
- **数据主权**: 各层级独立存储核心数据，不依赖上游
- **业务解耦**: 四层架构独立演进，降低系统耦合度
- **类型安全**: 全链路严格类型定义，杜绝隐式类型
- **可维护性**: 清晰的职责边界，便于长期维护扩展

## 五、核心设计原则与规范速查

### 5.1 架构原则
1. **领域解耦**: 严格按L1-L4划分业务边界，禁止跨层级JOIN
2. **写时复制**: 跨层级数据流转采用快照复制，下游独立修改
3. **类型安全**: 全链路严格类型定义，禁止any类型
4. **命名统一**: snake_case(Python) / camelCase(TypeScript)
5. **复杂度控制**: 优先原生方案，禁止不必要的第三方库

### 5.2 编码红线
- ❌ 严禁跨L层级JOIN查询
- ❌ 严禁使用any类型
- ❌ 严禁文件超过250行
- ❌ 严禁物理删除（仅逻辑删除）
- ❌ 严禁成功响应添加code/msg/data包装
- ❌ 严禁敏感信息明文存储
- ❌ 严禁生产环境开启调试模式

### 5.3 数据流转规则
```
L1 → L2: 线索创建时复制房源快照
L2 → L3: 项目立项时复制线索快照
L3 → L4: 营销发布时复制项目快照

规则:
- 仅复制字段名完全相同的字段
- 下游字段可独立修改
- 上游更新不自动同步下游
- 跨层级仅通过软引用关联
```

### 5.4 状态机定义
| 层级 | 状态字段 | 枚举值 |
|------|----------|--------|
| L1 | current_status | 在售/已售/下架 |
| L3 | current_status | 签约/装修/挂牌/已售 |
| L3 | current_renovation_stage | 拆除/水电/木瓦/油漆/软装/交付 |
| L4 | project_status | 在途/在售/已售 |
